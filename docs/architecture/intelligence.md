# GEO/AEO Scoring Engine & Discovery Intelligence Specification

This document formalizes the mathematics, algorithms, and pipelines driving the GEO/AEO scoring engine and the topic/prompt discovery system.

---

## 1. GEO/AEO Scoring Formulations

The platform computes four core metrics to assess answer engine presence:

### 1.1 Visibility Score
Tracks the raw percentage of brand mentions across all tracked response runs within a specific category, region, and period.

* **[CONFIRMED] Formula**:
  $$\text{VisibilityScore}(\text{brand}, \text{topic}, \text{period}) = \frac{\sum \text{runs where brand is mentioned}}{\text{Total runs for topic in period}} \times 100$$
* **[RECOMMENDATION] Commercial Intent Weighted Visibility**:
  To prioritize high-intent queries, each mention is weighted by its commercial intent score ($w_{\text{intent}}$ between $0.5$ and $2.0$), configured during prompt generation:
  $$\text{WeightedVisibility}(\text{brand}, \text{topic}, \text{period}) = \frac{\sum (\text{brand\_mentioned}_i \times w_{\text{intent}, i})}{\sum w_{\text{intent}, i}} \times 100$$

### 1.2 Share of Voice (SoV)
Compares target brand mentions against a defined competitive benchmark set.

* **[CONFIRMED] Formula**:
  $$\text{ShareOfVoice}(\text{brand}, \text{topic}, \text{period}) = \frac{\text{MentionCount}(\text{brand}, \text{topic}, \text{period})}{\sum_{e \in \text{CompetitorSet}} \text{MentionCount}(e, \text{topic}, \text{period})} \times 100$$
* **[CONFIRMED] Ranks**: Computed as an ordinal rank:
  $$\text{SoVRank} = \text{RANK}() \text{ OVER (PARTITION BY topic, period ORDER BY SoV DESC)}$$

### 1.3 Citation Impact Engine
Computes the authority and positioning quality of the sources cited by engines.

* **[INFERRED] Mathematical Formulation**:
  Based on peer-reviewed GEO research (Princeton/Georgia Tech, KDD 2024), citations earlier in a response carry higher prominence. The formula applies position-adjustment, domain authority, and exponential recency decay:
  $$\text{CitationImpact}(\text{domain}, \text{topic}, \text{period}) = \sum_{i \in \text{Citations}} \left( \frac{1}{\text{Position}_i} \times \text{AuthorityScore}(\text{domain}) \times e^{-\lambda t_i} \right)$$
  - $\text{Position}_i$: Character offset or link index in the response (earlier links have lower index values).
  - $\text{AuthorityScore}(\text{domain})$: Score ($0$ to $100$) representing domain trust (from licensed backlink indexes).
  - $\lambda$: Half-life decay factor (set to a 90-day half-life: $\lambda = \frac{\ln(2)}{90}$).
  - $t_i$: Age of the citation in days.

### 1.4 Model Weighting
Combines individual model visibility metrics into a single blended score based on monthly consumer usage market shares.

* **[INFERRED] Weighted Combination**:
  $$\text{BlendedVisibility} = \sum_{m \in \text{Engines}} \text{VisibilityScore}_m \times \text{UsageWeight}_m$$
  - Usage weights are updated monthly (e.g., ChatGPT: $0.62$, Gemini: $0.19$, Perplexity: $0.08$, Claude: $0.07$, Copilot: $0.04$). Blended values are always presented alongside granular engine filters.

## 2. The 4-Stage Prompt Discovery Pipeline

The discovery engine processes raw panel conversations ($1.5\text{B}+$ corpus) to surface relevant query clusters.

```
[1.5B+ Panel Corpus] ──> [Stage 1: Semantic Retrieval] ──> [Stage 2: Cross-Encoder Rerank]
                                                                    │
                                                                    ▼
[Canonical Selection] <── [Stage 4: Centroid Selection] <── [Stage 3: HDBSCAN Clustering]
```

### Stage 1: Semantic Retrieval from Panel
* **Input**: Seed keywords, brand domain, category.
* **Process**: Convert keywords to vector embeddings (using `text-embedding-3-large`). Run an Approximate Nearest Neighbor (ANN) query against the Qdrant database to retrieve panel conversations from the last 6 months, applying a cosine similarity threshold (>0.75).

### Stage 2: Filtering & Relevance Ranking
* **Process**: Candidate prompts are filtered by a cross-encoder model (e.g., `sentence-transformers/ms-marco-MiniLM-L-6-v2`) scoring the semantic similarity between the seed keyword and the retrieved prompt:
  $$\text{RelevanceScore} = \text{CrossEncoder}(\text{prompt}, \text{seed\_keyword}) \in [0, 1]$$
* This stage removes duplicate strings, handles translation of non-English queries, and filters out prompts containing PII or violating system policies.

### Stage 3: Clustering & Topic Extraction
* **Process**: Re-embed the filtered prompts. To capture user query behavior accurately, the system embeds the **original prompt** and its **query fan-out sub-queries** (internal searches generated by the engine) into a joint vector space.
* Cluster the combined embeddings using **HDBSCAN** (High-Density DBSCAN) with cosine distance metrics. This groups semantically related queries with varied phrasing (e.g., "best marathon running shoes" and "running sneakers reviews") into a single topic cluster.

### Stage 4: Canonical Prompt Selection
* **Process**: Within each cluster, a single canonical prompt is selected to represent the cluster. The selection maximizes both centroid proximity and real-world query volumes:
  $$\text{SelectionScore}_p = (1 - \text{CosineDistance}(v_p, v_{\text{centroid}})) \times \ln(\text{ConversationCount}_p)$$
* The prompt with the highest selection score is designated as the canonical prompt tracked by the visibility engine.

## 3. Architectural Summary Table

| Dimension | [CONFIRMED] Findings | [INFERRED] Systems Behavior | [ASSUMPTION] System Limits | [RECOMMENDATION] Optimization |
|---|---|---|---|---|
| **Scoring Formula** | Visibility represents mention counts divided by runs. | Citation impact is position-dependent; earlier citations score higher. | Static authority scores miss sudden trust drops on domains. | Refresh the domain authority cache every 15 days via external API feeds. |
| **Discovery Pipeline** | Processes seeds and categories to extract canonical tracking prompts. | HDBSCAN clustering processes parent prompts + query fan-outs. | Large-scale clustering over millions of vectors bottlenecks. | Perform nightly mini-batch centroid assignments; run full HDBSCAN weekly. |
| **Usage Weights** | Engine splits are exposed individually, and user limits scale by plan tier. | Blended metrics weight ChatGPT and Gemini higher based on usage traffic. | Hardcoded engine usage weights drift as market shares shift. | Update usage weights automatically via API feeds from traffic panels. |
| **Clustering Moat** | Fan-out sub-queries help group different phrasing formats together. | Sub-queries are extracted from live search grounding logs. | Scraping internal sub-queries fails when engines modify UIs. | Extract sub-queries from DOM elements or use statistical clustering. |
